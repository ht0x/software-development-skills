# Convert Skills into Plugin

Wrap one or more existing skills into a single installable plugin. The user already has skill content (a SKILL.md, possibly with `references/`, `scripts/`, or `assets/` subfolders) — the job here is packaging, not authoring: don't rewrite the skill's instructions or second-guess its content unless it's genuinely broken (missing frontmatter, wrong folder name, etc.).

## Why this is its own step

A plugin is a directory with a `.claude-plugin/plugin.json` manifest plus component folders (`skills/`, `agents/`, `.mcp.json`, etc.). A bare SKILL.md isn't installable on its own — it needs a manifest, a proper `skills/<name>/SKILL.md` path, and to be zipped into a `.plugin` file. This skill handles exactly that conversion, plus validation, so the user doesn't have to know the plugin schema.

## Workflow

### Step 1: Collect the skill file(s)

Ask the user (via AskUserQuestion if not already given in their message) how they want to supply the skill content:

- **Upload/attach** the SKILL.md file(s) directly in the conversation, or
- **Give a folder path** containing one or more skill directories (each with its own SKILL.md)

Both are valid — support whichever the user picks. If they give a folder path, list its contents (`ls`) and identify which subdirectories look like skills (contain a `SKILL.md` at their root, or are themselves a lone `SKILL.md` file with sibling `references/`, `scripts/`, `assets/` folders).

If multiple skill folders are found or uploaded, confirm with the user which ones to include in this plugin — don't silently bundle everything you find.

### Step 2: Validate each skill

For every skill being converted, check:

- A `SKILL.md` file exists and has YAML frontmatter delimited by `---`
- Frontmatter includes `name` (kebab-case, matches or will become the directory name) and `description` (third-person, with trigger phrases)
- The directory name matches the frontmatter `name` — if not, rename the directory to match (plugins require this alignment)
- Any relative links inside SKILL.md to `references/`, `scripts/`, or `assets/` resolve to files that actually exist in the folder

Report any problems found (missing frontmatter field, mismatched name, broken reference link) and ask the user how to resolve them — fix trivial issues yourself (e.g. renaming a folder to match `name`) but flag anything that changes the skill's actual behavior rather than silently editing it.

### Step 3: Ask for plugin name and category

Ask the user (via AskUserQuestion, one question at a time or combined if natural):

1. **Plugin name** — must be kebab-case (lowercase letters, numbers, hyphens only). If the user gives something else (spaces, capitals, underscores), convert it and confirm the converted form with them rather than guessing silently.
2. **Category** — a free-text category/theme for the plugin (e.g. "productivity", "game-dev", "documentation"). Plugin manifests have no native `category` field, so this gets stored in `plugin.json`'s `keywords` array. Mention this mapping to the user in passing so it's not a surprise later.

Do not ask for version, author, or a plugin-level description at this stage — fill those with sensible defaults (version `0.1.0`, author from context if known, description auto-summarized from the bundled skill(s)) and show them to the user as part of the Step 4 confirmation, rather than interrogating for them upfront.

### Step 4: Confirm before generating

Before writing any files, show the user a short summary and get explicit confirmation:

```
Plugin: <plugin-name>
Category (keywords): <category>
Version: 0.1.0
Description: <auto-generated one-liner>
Skills included:
  - <skill-name-1> — <one-line summary from its description>
  - <skill-name-2> — <one-line summary from its description>
```

If the user wants to change the auto-filled description, version, or author, adjust now. Only proceed to Step 5 after they confirm.

### Step 5: Build the plugin structure

Create the plugin directory:

```
<plugin-name>/
├── .claude-plugin/
│   └── plugin.json
├── skills/
│   ├── <skill-name-1>/
│   │   ├── SKILL.md
│   │   └── ... (references/scripts/assets, copied as-is)
│   └── <skill-name-2>/
│       └── SKILL.md
└── README.md
```

`plugin.json`:

```json
{
  "name": "<plugin-name>",
  "version": "0.1.0",
  "description": "<auto-generated one-liner>",
  "keywords": ["<category>"],
  "author": {
    "name": "<author, if known>"
  }
}
```

Copy each validated skill folder into `skills/<skill-name>/` unchanged — preserve `references/`, `scripts/`, `assets/` subfolders exactly as they were. Do not alter SKILL.md content beyond the fixes already confirmed with the user in Step 2.

Write `README.md` with:

1. **Overview** — what the plugin does (derived from the skill(s) it bundles)
2. **Components** — list each included skill and its trigger description
3. **Setup** — note none required, unless a bundled skill's scripts have dependencies worth flagging
4. **Usage** — how each skill triggers

### Step 6: Validate structure

Before packaging, verify:

- `.claude-plugin/plugin.json` exists and is valid JSON with at least `name`
- `name` is kebab-case
- Every folder under `skills/` contains a `SKILL.md`
- Each `SKILL.md` still has valid frontmatter after copying

Report pass/fail for each check. Fix anything that fails before packaging.

### Step 7: Package and deliver

Zip the plugin directory into a `.plugin` file:

```bash
cd /path/to/plugin-dir && zip -r /tmp/<plugin-name>.plugin . -x "*.DS_Store" && cp /tmp/<plugin-name>.plugin /path/to/outputs/<plugin-name>.plugin
```

Always stage the zip in `/tmp/` first, then copy to the outputs folder — direct writes to the outputs folder can fail due to permissions.

Present the resulting `.plugin` file to the user with the file-sharing tool available in the environment (e.g. `present_files`). The file renders as a rich preview the user can browse and install with one click.

## Edge cases

- **User uploads a skill with no bundled resources, just a bare SKILL.md**: still goes through the same `skills/<name>/SKILL.md` structure — no references/scripts/assets folders needed.
- **Two uploaded skills have the same `name` in frontmatter**: flag the collision immediately and ask the user to rename one before proceeding — plugin skill directories must be unique.
- **Folder path contains non-skill files/folders alongside real skills**: ignore them, but mention what was skipped and why (no SKILL.md found).
- **User wants to add this skill's output to an *existing* plugin rather than create a new one**: ask whether an existing plugin directory should be extended (add to its `skills/` folder and merge `keywords`) instead of creating a fresh plugin — don't assume a new plugin is always wanted if they mention a plugin name that sounds like it might already exist.
